# 启动慢与卡顿根因分析 · 进程/线程解决方案（v1.14.0）

> 回答三个问题：①启动为什么慢？②运行为什么卡？③是不是功能太多了、
> 要不要换语言重写？结论先行：**慢和卡的根因不是"功能太多"，而是
> "采样与 UI 挤在同一条主线程"+"启动期的 import 成本"**。v1.14.0 已用
> 独立采样线程根治卡顿与漏检；启动慢属 Python import 机制固有成本，
> 已用后台加载+加载页缓解，进一步压缩见 Nuitka 方案
> （`docs/Golang重写方案调研.md` 方案 D）。

---

## 一、启动慢：时间花在哪里？

| 阶段 | 耗时量级 | 说明 |
| --- | --- | --- |
| `import mediapipe` + `import cv2` | **2~5s** | 两库共载入数百个模块与原生 DLL，是启动耗时的大头 |
| 打开摄像头 `VideoCapture(0)` | 0.5~2s | 驱动初始化，机器差异大 |
| 加载 `.task` 模型 ×2 | 0.5~1s | face_landmarker + pose_landmarker |
| tkinter UI 构建 | <0.3s | 很快，不是瓶颈 |
| Ollama 探测/拉起（后台） | 0（异步） | 已在后台线程，不阻塞 |

**已做的缓解**（v1.11.0/v1.13.0）：视觉后端全部搬进后台线程加载，
UI 秒开 + 启动加载页（splash）占位；LLM 预热、台词池延迟 20s 错峰。
因此用户感知的"启动慢"= 加载页那几秒，而非白屏等待。

**进一步压缩的正确路径**：不是换语言——检测内核（MediaPipe/OpenCV）
本就是 C++，换 Go/Rust 提速的只有壳；最低成本是 **Nuitka 编译**
（实测同类项目启动 -34%、体积 -43%，2~4 天工作量），详见
`docs/Golang重写方案调研.md`。

## 二、运行卡顿：不是功能太多，是"谁在主线程干活"

v1.13.0 及以前，单进程实时模式的每一次 `tick()`（由 UI 主循环驱动）里
串行做了：**摄像头读帧 → MediaPipe 双模型推理 → 眨眼判定 → 指标统计 →
UI 全量重绘**。这条链有三个连锁恶果：

1. **推理卡 UI**：单帧推理 30~80ms 全部落在 tkinter 主线程，粒子背景、
   桌宠动画肉眼掉帧；
2. **UI 卡采样**：反过来，UI 忙（粒子多、建议区每秒 15 次全量重建 Label）
   时 tick 变慢 → 实际采样率从 15fps 掉到个位数——**快速眨眼只有
   100~300ms，低帧率下只覆盖 1 帧，旧判据「连续 2 帧闭眼」直接漏检**。
   这就是"之前精度够、功能越加越多之后不行了"的真相；
3. **缓冲滞后**：OpenCV 默认缓冲多帧，读取慢于相机帧率时拿到的是
   0.3s+ 前的旧帧，卡顿进一步放大检测延迟。

另外两个"看起来像卡顿"的独立问题：

- **距离/颅椎角不实时**：旧版界面显示的是 300 帧（约 20 秒）滚动均值，
  坐姿变了要十几秒才反映——这是统计口径问题，不是检测精度问题；
- **建议文字闪烁**：建议区每次刷新都销毁全部控件再重建（15 次/秒），
  tkinter 来不及双缓冲，视觉上一闪一闪。

## 三、v1.14.0 的解法（线程为主，进程为辅）

| 措施 | 位置 | 效果 |
| --- | --- | --- |
| **独立采样线程** `_CameraSampler` | `core/monitor.py` | 采集+推理+眨眼判定按 `fps_target` 固定节奏在专线程运行；样本入有界队列，`tick()` 批量取走逐帧入指标——**UI 卡顿不再造成漏检，推理不再拖累 UI**；线程崩溃自动回退模拟器 |
| **眨眼判据加墙钟兜底** | `core/blink_counter.py` | 帧数判据(≥2帧)之外，闭眼墙钟时长 ≥0.10s 也计一次眨眼——即便偶发掉帧，快速眨眼也不漏 |
| **CAP_PROP_BUFFERSIZE=1** | `monitor.py` / `vision/worker.py` | 摄像头缓冲压到 1 帧，永远处理最新画面，消除 0.3s+ 滞后 |
| **距离/CVA 改短窗实时值** | `core/metrics.py` | 界面与建议改用近 3 秒墙钟窗均值（`recent_*`），姿势变化秒级上屏；20s 均值仅保留给会话报告 |
| **建议区差量渲染** | `ui/app.py` | 内容签名不变则完全不动控件，闪烁根除 |
| （已有）`--mp-vision` 子进程 | `vision/worker.py` | 需要更强隔离时把采集+推理放独立进程，彻底绕开 GIL |

### 线程 vs 进程怎么选？

- **默认（本版）：线程**。MediaPipe 推理在 C++ 层会释放 GIL，采样线程
  与 UI 线程真并行，开销小、无跨进程序列化成本、崩溃回退简单；
- **`--mp-vision`：进程**。Python 层计算占比高或低配机上仍掉帧时启用，
  子进程独占摄像头与推理，Queue 只回传轻量样本；
- **不建议**：把 UI 也拆进程/换语言重写——比赛期收益低风险高，
  产品化阶段再按 `docs/Golang重写方案调研.md` 走 Go+Wails 路线。

## 四、结论

- 启动慢 ≠ 功能多：是 mediapipe/cv2 的 import 固有成本，已用后台加载+
  加载页兜住体验，要再快用 Nuitka，不必换语言；
- 卡顿/漏检 ≠ 功能多：是采样与 UI 同线程互相拖累，v1.14.0 独立采样
  线程后两者解耦，功能照留、精度照旧；
- 实时性与闪烁是两个独立的统计口径/渲染问题，已分别用短窗实时值与
  差量渲染修复。
