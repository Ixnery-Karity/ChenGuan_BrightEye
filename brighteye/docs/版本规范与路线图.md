# 宸观 BrightEye · 版本规范与路线图（v1.16.0 制定）

> 目的：统一版本号语义、梳理既有版本脉络、规划未来版本，
> 让「比赛演示 → 软件成品 → 产品化」的演进路径清晰可依。

---

## 一、版本号规范（自 v1.16.0 起执行）

格式：**`主版本.次版本.修订号[-阶段]`**（语义化版本，SemVer 裁剪版）

| 位 | 何时递增 | 示例 |
| --- | --- | --- |
| **主版本** | 产品形态级变化（技术栈迁移 / 商业形态落地） | 1.x 比赛演示期 → 2.x 产品化 |
| **次版本** | 新增功能、能力升级（每次功能迭代 +1） | v1.15 → v1.16 |
| **修订号** | 只修 bug / 文档，不加功能 | v1.16.0 → v1.16.1 |
| **阶段后缀** | `-demo` 比赛演示版 → `-rc` 候选发布 → 无后缀=正式版 | v1.16.0-demo |

**发版纪律**（每次发版四件套，缺一不算发布）：
1. `config.py` 版本号更新；
2. `CHANGELOG.md` 登记（修复/新增/变更分节）；
3. `README.md` 与 `docs/功能清单.md` 同步；
4. 源码 zip 入 `challenge/版本控制/`（带版本号，保留旧包不覆盖）+ git 提交推送。
   自 v1.16.0 起追加：**git tag**（`git tag v1.16.0-demo`），GitHub 上可直接检出任意历史版本。

## 三、未来版本规划

依据：已完成功能盘点（`docs/功能清单.md`）+ 已立项未实施的方案
（皮肤系统 `docs/皮肤系统设计方案.md`、UI 升级 `docs/UI技术栈升级路线.md`、
Nuitka/Go `docs/Golang重写方案调研.md`、Live2D `docs/桌宠技术栈调研_天选姬.md`、
情绪 ML 升级 `docs/大模型接入与部署指南.md` 第六节）。

### 1.x 线 —— 比赛期（Python + tkinter，离线铁律不破）

| 版本 | 主题 | 内容 | 状态 |
| --- | --- | --- | --- |
| ~~v1.17.0~~ | 皮肤系统（一） | AI 生成弥悠官方皮肤 + skins/ 目录隔离加载 + 上传规则；安装包大模型集成/防版本冲突/Release 制 | **已发布 07-21** |
| **v1.18.0** | 皮肤系统（二） | 界面内上传立绘 + soul.md 皮肤包（含人格切换）；主题随皮肤联动 | 方案就绪 |
| **v1.19.0** | 体验打磨 | 情绪识别可选 ML 升级（FER2013+blendshapes 特征，`emotion_clf.pkl` 软加载，保留 FACS 兜底）；报告一键导出 PNG/PDF 进 PPT | 设计已预留 |
| **v1.20.0** | 分发增强 | Nuitka 编译（体积-43%/启动-34%，调研结论 D 方案） | 调研就绪 |
| v1.x 收口 | 赛前冻结 | 只出修订号（-rc），答辩包定版 | — |

### 2.x 线 —— 产品化（赛后，技术栈升级）

- **v2.0**：UI 迁移（PySide6 或 Tauri，二选一按 UI 升级路线文档）；
  桌宠独立进程 + Live2D 动态立绘（WebSocket 读监测数据，跨语言解耦）；自动更新器。
- **v2.1**：多端正式化——手机 App（复活 Android 线）经局域网/云中转同步；家长/自我周报推送。
- **v2.2**：账号体系与云端可选备份（默认仍本地，隐私锁卖点不丢）。

### 3.x 线 —— 商业化（对齐商业计划书四层收入）

- B 端管理台（学校/眼科诊所/企业工会批量部署与聚合看板）;
- C 端订阅（皮肤/人设包/高级报告）；硬件联动（屏幕挂灯/升降桌 DDC 之外的 IoT）。
